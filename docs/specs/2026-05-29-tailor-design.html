TAILOR

Job-specific resumes, cut only from cloth you actually own.

Design spec · 2026-05-29 · author: cmcmath89 · status: approved for planning · stack: React → Next.js frontend, Node/Claude backend

§Thesis

People who apply to many jobs re-tailor their resume by hand for every posting — slow, repetitive, and error-prone. TAILOR removes that work. You upload your real career corpus once — resumes, role variants, achievement memos, LinkedIn export, portfolio — and for every job description, TAILOR reads everything, surfaces the matches you'd have missed, phrases them to line up with what the employer is asking for, and shows you a tailored, export-ready resume with a fit score.

The product's whole promise is trust: every line is grounded in what you've actually done. Where the job asks for something you haven't named explicitly, TAILOR may infer it — but only when it can be honestly derived from your experience (you know Tableau, the job says "data visualization" — that's a fair, defensible claim). The inference yield is what widens your match to each posting without ever inventing experience you don't have.

"A tailor alters the cloth you bring. They never weave fabric you don't own." — the grounding rule, stated as the product's creed.

1Architecture

The frontend is a thin, fast UI over a backend that owns everything sensitive: document parsing, the evidence store, all Claude calls, scoring, and export rendering. The frontend never holds the Claude key or parsing logic, so the Next.js app can be built later against a stable API contract (§9).

Frontend ⇄ Backend over JSON; backend is the only thing that touches Claude, the DB, and storage.

2Data Model

Two layers that never mix — this separation is the trust guarantee (see §4). The corpus is immutable ground truth; a tailoring is a disposable, derived view.

The Cloth + The Form — immutable

corpus_document · evidence_unit · canonical_profile

  • corpus_document — one uploaded file. Raw bytes + parsed text + source metadata.
  • evidence_unit — the atom of trust ("Thread"). A discrete real claim — "Built Tableau dashboards for the sales team, 2021–2022" — deduped & merged across overlapping docs. Links to source doc(s) + location.
  • canonical_profile — "The Form" (§4). The corpus normalized into one profile: canonical threads, resolved entities/titles/dates, and grouped skill variants. Facts only — no stored relationships; the model infers those at match time. Built once, reused for every Fitting.

Inferences are never written back here. Evidence units are facts about you; an inference is not a new fact.

The Fitting — derived

job · tailoring · resume_bullet · transformation

  • job — a parsed JD: extracted requirements, skills, keywords, seniority.
  • tailoring — one resume for one job: selected bullets, fit score, template, edits. Saved & revisitable.
  • resume_bullet — a line in the output. Carries its transformation type + the evidence_unit id(s) it derives from.
// the load-bearing relationship
evidence_unit  1 ──< resume_bullet  >── 1 transformation_type
                          │
                          └── provenance: { evidence_ids[], relationship, type }

// invariant enforced server-side, every bullet, no exception:
bullet.evidence_ids.length > 0   // a bullet with no thread cannot exist

3The Pipeline

From raw uploads to a finished Fitting. Steps 1–2 happen once per corpus / once per job; steps 3–7 run each time a Fitting is generated.

StepWhat happens
1 · Ingest + canonicalizeUpload many docs → parse → split into threads, then canonicalize (§4): dedup/merge restated facts, resolve entities, group skill variants → the Form. Done once per corpus.
2 · Read JDPaste/parse the Pattern → Claude extracts required skills, responsibilities, keywords, seniority.
3 · MatchFind threads on the Form that already directly satisfy Pattern items.
4 · Infer (yield)The model reasons over the whole Form against the Pattern to surface defensible matches the user never listed (§5) for unmet items — the inference yield. Ungroundable reaches are never proposed.
5 · VerifyServer-side gate (§5): is each bullet entailed by its cited threads? Drop anything that asserts more than the cloth supports.
6 · ComposeAssemble a single-column, ATS-safe resume; phrase to mirror Pattern language while staying true to threads. Compute the Fit score.
7 · PresentOpen the Fitting Room (§6): preview, what-changed, provenance, edit, export.

4The Canonical Career Profile — "The Form" · moat #1

A real career corpus is messy: three overlapping resumes, the same achievement worded five ways, skills scattered across a portfolio and a LinkedIn export, inconsistent titles and date formats. Matching raw text against a JD wastes that richness. So before any job is ever seen, TAILOR canonicalizes the corpus into one normalized profile — the Form.

Why this is a moat. Competitors re-parse a single resume per application. TAILOR accumulates a canonical, unified profile of everything you've ever done. It is built once and compounds — every doc you add makes every future application better. The Form is a durable user asset, not a throwaway parse.

What canonicalization produces

StepWhat it doesExample
Dedup & mergecollapse restatements of the same fact into one canonical thread, keeping every source link"Built Tableau dashboards" (resume A) + "Owned sales reporting in Tableau" (memo) → one thread, two sources
Entity resolutionunify employers, titles, and dates across docs into a consistent timeline"Sr. Analyst" / "Senior Data Analyst" / "Data Analyst II" → one canonical role
Skill groupingcollapse obvious surface variants of the same skill (no curated ontology, no taxonomy to maintain)"Tableau Desktop", "tableau", "TBL dashboards" → one skill
Facts only — no relationship graph. The Form deliberately stores what you've done, not how skills relate to each other. We do not precompute a competency graph or seed an ontology — that would be brittle and would cap the reach to whatever edges we hand-encoded. Judging that "Tableau entails data visualization in this context" is left to the model at match time (§5), where it's done fresh against each Pattern.

The Form is what every Fitting is built on, and what the Atelier persists. It is part of the immutable ground-truth layer (§2): canonicalization organizes the cloth, it never adds to it.

5Inference Yield — "The Reach" · moat #2

Inference yield is the product. It is the set of matches TAILOR surfaces that the user would never have put on their resume themselves — defensible claims derived from the context of their own documents, lined up against what the employer is asking for. A user knows Tableau; the job asks for "data visualization"; the user never thought to phrase it that way. TAILOR closes that gap automatically.

How yield is generated

Yield comes from the model reasoning directly over the whole Form against the Pattern — proposing the defensible generalizations and synonyms that connect what you've actually done to what the JD asks. We deliberately do not precompute a competency graph or ontology (§4); the model is far better at judging entailment in context, and a fixed graph would only cap the reach. The honesty guarantee is therefore not a graph — it is grounding (§6) and the verification gate (§7). With no structure constraining the model, those two carry the full weight in this design, which is why they're the most important things to test (§14).

The model's reasoning, not a stored structure: skills you listed → competencies they entail → requirements you never explicitly claimed. Yield = the connections it surfaces.

Yield is measured and surfaced

Because it's the core value, yield is a visible number, not a hidden mechanic. The Fitting Room (§8) reports it: "14 matches surfaced — 9 you'd never listed." Over time the Atelier can show yield per application, so the user sees TAILOR doing the work they'd otherwise do by hand.

Yield maximizes reach; grounding (§6) and verification (§7) keep it honest. These two forces are in deliberate tension — push reach as far as the cloth truthfully allows, and not one thread further.

6Grounding & Alterations — keeping the reach honest

"Every bullet must trace to evidence" sounds like it forbids inference. It doesn't — if inference is a typed, bounded transformation of cloth you already own rather than free invention. The resolution is two rules:

Two layers never mix. The corpus is immutable ground truth. A tailoring is a derived view — every bullet on it is a transformation of one or more evidence units, with the transformation type recorded. Inferences are never written back into the corpus.

Every bullet is exactly one of four typed transformations

TypeWhat it isGrounding ruleTableau example
Verbatimrestates an evidence unittrivially grounded"Built Tableau dashboards for sales"Allow
Rephrasesame fact, JD-aligned wordingasserts no fact not in the evidence"Designed Tableau dashboards to drive sales decisions"Allow
Infera broader competency the evidence entailsmust be an entailment: IS-A / tool→competency / synonym — never mere associationTableau → "data visualization"Allow + flag
Composeone claim synthesized from several evidence unitsmust follow from the cited set togetherTableau + Looker + SQL → "data analytics"Allow + flag

What is banned, always

Banned alterationWhy
Inventing metrics / numbers ("improved sales 30%")not entailed by any threadBan
Inventing projects, employers, titles, or datesfabricates clothBan
Adding responsibilities the corpus doesn't supportassociation ≠ entailmentBan
Inferring an adjacent tool you never used (Tableau → "Power BI")sibling, not a generalizationBan
An alteration may widen or restate what the cloth says. It may never add to it. If you used Tableau, you did do data visualization — that claim is defensible in an interview, which is the test every inference must pass.

7The Verification Gate

Grounding is enforced by the system, not trusted to the model. Generation and verification are separate passes so the model that proposes a bullet is not the only thing judging it.

  1. Propose. When Claude proposes a bullet it must emit a structured object: type, evidence_ids[], and for inferences the relationship ("Tableau is a data-visualization tool").
  2. Verify. A separate server-side pass checks each bullet: Is this claim entailed by the cited threads via the stated relationship? Does it assert anything the threads don't? Pass → keep. Fail → drop (and log for transparency).
  3. Hard invariant. Independently of the model, the backend rejects any bullet with zero evidence_ids, or any number/metric not literally present in its cited threads. The model cannot bypass this.
// structured proposal the model must return per bullet
{
  "text": "Designed data-visualization dashboards to drive sales decisions",
  "type": "infer",
  "evidence_ids": ["ev_8f2"],
  "relationship": "tool_to_competency: Tableau ⊂ data visualization"
}
// → verifier asks Claude: entailed? asserts nothing extra? → {ok:true}
// → hard check: any digits in text present in ev_8f2? → none added → pass

This gate is the single most important test target (§14). If it leaks, the product's core promise breaks. Treat "an ungrounded bullet reached the user" as a sev-1 class of bug.

8The Fitting Room — results modal

After generation ("disbursement"), the user lands in the Fitting Room — a modal that lets them see, trust, adjust, and export without leaving the flow. Speed is the point: inferences are already applied; the user only intervenes if they want to.

Template preview

Live render of the resume in the chosen template. Switching templates re-renders instantly; never changes content, only presentation (and all templates stay ATS-safe).

Fit score panel §9

Score bar + percentages: overall fit, keyword coverage, requirement coverage, format parseability. Updates as the user edits.

What the AI changed yield

Leads with the inference yield headline — "14 matches surfaced, 9 you'd never listed" — then lists added / rephrased / inferred bullets. Inferred and composed bullets are visually marked, and each shows "where the thread came from" — its source doc / evidence + relationship. Provenance is never hidden.

Inline edit & quick-remove

Edit any bullet in place. Every inference has a one-click remove (it was auto-included, so dropping it is lightweight — never a blocking approval step). Edits re-score the Fit live.

Export

One action, user picks the format: ATS-safe PDF or DOCX. Both render server-side from the same composed resume so they're identical in content. The finished file is "ready to wear" and saved to the Fitting for re-download later.

9The Fit Score (ATS)

A single headline percentage with a breakdown, shown as bars in the Fitting Room. Deterministic given a (resume, job) pair so it's testable and reproducible — Claude extracts the Pattern's requirements/keywords, but the scoring math is plain code.

Overall fit84%
Keyword coverage78%
Requirement coverage88%
Format parseability96%

10The Atelier — workspace

The reason TAILOR beats hand-tailoring for the repeat applicant: do the heavy lifting once, reuse forever.

  • Accounts. The Cloth is uploaded once and the Form (§4) is built once and stored, ready for any number of Patterns.
  • Per-job Fittings. Each new JD creates a saved Fitting — revisit it, re-export, or duplicate it as a starting point.
  • Re-run on a richer Form. Add a new achievement memo → the Form grows → re-generate any Fitting and yield typically rises.
  • Dashboard. Every Pattern + its Fittings in one studio, with fit scores and inference yield at a glance.

16The Best-Version Guarantee — what "done" means

When a Fitting finishes, it must be the best defensible version of the user's resume for that Pattern given their Cloth — good enough to submit with confidence. Operationally: it is impossible for the Fitting Room to suggest an improvement the corpus could have supported. Any "to improve, add X" item must be a genuine gap in the user's experience, never a thread TAILOR failed to surface.

The guarantee is structural, not iterative. The improvement list shown to the user is computed as a set difference:
improvements = {Pattern requirements} − {requirements the Cloth can defensibly cover}
If coverage is computed exhaustively against the Form before composing (§4–§5), the only items that can remain are requirements no thread supports — real gaps ("the role needs an AWS certification you don't have", "asks for 5 yrs, your timeline shows 3"). A corpus-supportable keyword can never appear there, because anything supportable was already pulled in or only displaced by the length budget (§17).

The bounded coverage loop

The guarantee is enforced by a tight loop over a deterministic, computable objective — requirement/keyword coverage — not over subjective writing quality:

  1. Plan. Map every Pattern requirement → the evidence units that entail it (direct hit or defensible alteration, §6). This is the coverage map, built before any prose.
  2. Generate. Compose from the selected threads (§17), instructed to express every covered requirement.
  3. Coverage-diff deterministic. Recompute coverage on the produced draft. Any requirement that is corpus-supportable but absent from the draft is a defect — exactly the failure mode this section forbids.
  4. Targeted revise. Feed only the specific missing-but-supportable requirements back as a narrow instruction ("add evidence for X; change nothing else").
  5. Stop at the fixed point. Halt when the coverage-diff is empty, or the last revision closed zero new gaps. Typically 2–3 rounds.

Not open-ended self-refinement. "Make it better" loops are documented to degrade output — models lock onto a draft and pad it, and self-critique only helps when the model's error rate is already near-zero. The loop here only ever closes measurable coverage gaps; it never rewrites for taste. The coverage-diff is the objective; the verification gate (§7) is the wall that stops the loop from inventing evidence to "close" a gap.

A Fitting is done when every defensible thread that matches the Pattern is on the page, and every remaining gap is one the cloth genuinely cannot cover. Nothing left to honestly add.

17Selection & Ranking — fitting to the length budget

A deep corpus (e.g. a 9-yr backend engineer, 5 roles) yields 40–60 defensible candidate bullets for a single Pattern, but a resume holds ~18–24. Choosing which threads make the page — and in what order — is its own step, distinct from grounding (§6, which decides whether a bullet is allowed) and from the Fit score (§9, which grades the finished resume).

Selection = monotone submodular maximization under a knapsack (length) constraint. This is the same structure as extractive summarization and RAG context selection, with a provable (1−1/e) greedy guarantee. The objective is weighted coverage of Pattern concepts; submodularity encodes diminishing returns, so the second bullet proving "Kubernetes" counts for less than the first — redundancy is handled by the objective, not by ad-hoc penalties.

The selection model

  • Objective: maximize covered Pattern concepts, weighted by (a) Pattern priority — required qualifications ≫ nice-to-haves — and (b) role recency, so coverage delivered through a recent role is worth more (ATS weights recency, §9).
  • Algorithm: density-greedy — pick the bullet with the highest marginal coverage-gain per line consumed, repeat until the budget is full. No model in this loop; it is plain, deterministic code, testable like the Fit scorer.
  • Seniority emphasis: line-cost scales with role age — the current role gets the largest budget; roles 4–5 cap at 1–2 bullets. Falls out of the knapsack cost-weighting and matches recruiter norms.
  • Ordering is convention, not optimization: roles newest-first, bullets within a role by coverage contribution.

Length budget

Decision (v1): up to two pages, substance-driven for senior/deep profiles — budget ~18–24 bullets — collapsing to one page when the corpus can't fill a second page without padding. (Senior two-pagers measurably out-pull one-pagers when page two is real content; the limiting factor is the human reader, not the parser.) The Fit score's format component (§9) keeps it ATS-safe regardless.

Relationship to §16: selection is where the length budget bites. If a corpus-supportable requirement is left uncovered, the coverage-diff (§16) checks whether swapping a low-value bullet would cover it; only requirements that lose that trade-off — or have no thread at all — are honestly reported as gaps.

18Model Routing — which model does what

Different pipeline stages have different cost/quality profiles. The routing below is the v1 default (verified mid-2026); model names will drift, so treat the roles as stable and the named models as swappable.

StagePrimaryCheaper fallbackWhy
Parse + canonicalize (§4)Gemini 3 Flash (1M ctx)Flash-Lite / Haiku 4.5high-volume structured JSON, doc-native, cheap
Match + infer (§5)Claude Opus 4.8 / Sonnet 4.6 — in-context over the whole Form, no embeddings~90 threads fit in context; more accurate than retrieval at this scale (vectors deferred — see Open Q)
Generate / compose (§16)Claude Opus 4.8Sonnet 4.6 (~40% cost)reasoning + steerable honesty; customer-facing output
Verification gate (§7)GPT-5.5 — different vendorGemini 3.1 Proindependence: a same-family judge shares blind spots and rationalizes the same fabrications
The one non-negotiable split: the verifier is a separate pass, ideally a different vendor than the generator. A model grading its own output in one pass is theater — it is the brittle self-correction §16 warns against. Cross-vendor decorrelates failure modes; the ops cost is a second API key. If consolidating, the minimum is a separate stateless pass that sees only (claim, cited evidence) — never the generator's reasoning.

Don't over-engineer. In an MVP one strong model can handle parse + generate together; the verifier is the only mandatory separation. Specialize the cheap high-volume extractor and the frontier generator only once unit economics or quality demand it.

Privacy: the corpus is dense career PII. Use zero-retention API tiers; for the highest-sensitivity deployments, self-host the extractor/embedder so profile text never leaves the boundary. (Ties to §13.)

19Seed Test Fixtures — how we test the guarantee

The moat (§4–§5) and the guarantee (§16) are only as trustworthy as what we test them against. Fixtures are LLM-synthetic, with authored ground-truth labels — seeded for realism from permissively-licensed real datasets (CC0/MIT) but never carrying real PII into our systems. This gives privacy safety and, critically, match labels we control.

Recipe

StepWhat it produces
1 · Truth file per personaOne canonical YAML/JSON: 5 dated roles, each with 3–5 atomic achievements carrying hard attributes (metric, tech, scope). This is the label source of truth.
2 · Fan out 8–12 docsLLM-generate overlapping variants from that one truth file — terse 1-pager, 3-pager, LinkedIn prose, an older resume missing the last job, a brag-doc, a cover letter. Prompt constraint: reword, never invent — every claim traces to a truth-file attribute.
3 · Paired PatternsPer persona, JDs at three tiers: Good (target role), Potential (adjacent), No-Fit (cross-domain).
4 · Defensible-match labelsCriterion-level rows: which truth-file attribute satisfies each Pattern must-have, with a span pointer.

The two oracles these give us

  • Canonicalization oracle: dedup (§4) must collapse all phrasings of attribute X across the 8–12 docs into one thread with all source links.
  • Best-version oracle: for each (persona, Pattern), the finished Fitting must surface the strongest phrasing of every attribute that matches a must-have — directly asserting §16.

Persona roster spans a 9-yr backend SWE (flagship), a mid full-stack dev, a data→backend switcher, a sparse career-changer (tests inference), and a non-SWE control (cross-domain No-Fit). Inject hard cases — employment gaps, over-qualification, prompt-injection in a skills list — so guardrails are exercised, not just the happy path.

Stay agnostic. The product must never be tuned to make any one profile — including the author's own resume — score better. That is invisible bias that degrades every other user and invalidates these fixtures. The author's resume is just one more test profile, never the target. Validate every change against the diverse roster, not one corpus.
